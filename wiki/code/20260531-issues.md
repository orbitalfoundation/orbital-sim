# issues to look at - june 1 2026

# Conserving memory

The hardest problem is memory per simulation instance. If every new browser session spins up a fresh bus with GEBCO loaded, you hit memory limits fast at any scale. The solution is to treat read-only data (GEBCO raster, land cover) as process-level singletons shared across all sim instances, and only the mutable tick state is per-session. Your bus architecture actually supports this cleanly — bus.elevation could be installed once at process startup and injected into any new bus instance rather than reloaded. Worth designing for explicitly before writing server code.

# publishing state

The server is genuinely shallow: sim-manager calls createBus() and bus.resolve({load, run}), the server just wires the EventEmitter to socket.io rooms. All domain logic stays in @orbital/bus and @orbital/agents.

One thing worth noting as a design decision when you come back to it: right now tick events only carry {tick, t, dt} — timing, not field data. A scenario HTML page that wants to render a heatmap will need a way to get field values too. The clean path is an emitter agent in @orbital/agents that samples bus.insolation, bus.elevation etc. on each tick and fires an { emit: {...} } event — the server forwards that to the room the same way it forwards tick. That agent would live in the agents package, not the server, keeping the boundary clean.

Two distinct questions worth separating:

Memory isolation — mostly yes, but same process

Each createBus() creates its own bus instance with its own agent objects and internal event routing. Two sims don't share bus state. But they're all running in one Node.js process — same V8 heap, same event loop thread. A CPU-heavy sim blocks them all; a memory-leaking sim starves them all. There's no OS-level isolation.

One subtle exception: module-level state in agent files is shared. The GEBCO elevation tile cache in packages/agents/elevation.js lives at the module scope — it's loaded once and shared across every sim that uses it. That's actually fine (and efficient) as long as that state is read-only after load. If an agent ever mutates module-level state you'd get cross-sim contamination.

Lifecycle — currently none

You're right. Right now:

Sims run until DELETE /api/sim/:id is called explicitly
Server restart wipes the sims Map entirely (no persistence)
A user who navigates away or closes the tab leaves their sim running forever
Socket.io room becomes empty but the tick loop keeps firing at full hz into the void
The natural policy that fits your use case is probably:

Idle timeout: stop a sim after N minutes with no socket subscribers
Max age: hard cap (e.g. 24h) for long-running scenarios
Session-linked: optionally tie a sim's lifetime to the auth session that created it
