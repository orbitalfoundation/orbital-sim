## Claude Comments: What was built

**`packages/agents/atmosphere.js`** — single-layer energy balance model (EBM).

The physics: `C · dΔT/dt = F(CO₂) − λ · ΔT`

- `F = 5.35 · ln(CO₂/280)` W/m² — Myhre 1998 forcing formula
- `λ = 1.24` W/m²/K — climate feedback parameter, gives ECS = 3.71/1.24 ≈ 3°C
- `C = 3×10⁸` J/m²/K — effective ocean mixed-layer heat capacity
- τ = C/λ ≈ 7.7 years — characteristic response time

SSP CO₂ trajectories from RCMIP / AR6 WG1 Table 7.SM.6, stored as decadal
breakpoints and interpolated at each tick. Four scenarios: SSP1-2.6, SSP2-4.5,
SSP3-7.0, SSP5-8.5.

Spin-up: on `registered`, the model runs silently from 1850 to `this.t0` using
a historical CO₂ record, so the starting state reflects committed warming
(~1.6°C by 2024 in the model; slightly high vs. observed ~1.3°C — likely
due to absence of aerosol cooling term, see TODOs).

The agent seeds `bus.time` on `registered` so the correct year is recoverable
from `event.t` regardless of whether the bus was started via CLI (`--t0` flag)
or via the web server (no `tStart`).

**Geographic pattern scaling** — implemented as a 360×180 Float32Array built
lazily on first tick, sampling `bus.elevation` for land/sea at each 1° cell.
Amplification factors derived from AR6 WG1 Chapter 11 / Atlas:

| Zone | Factor |
|---|---|
| Arctic (lat > 65°N) | 3.2× |
| Antarctic (lat > 65°S) | 1.7× |
| High latitude land (50–65°) | 2.0× |
| High latitude ocean (50–65°) | 1.3× |
| Mid-latitude land (30–50°) | 1.5× |
| Mediterranean / MENA land | 1.5 × 1.2 = 1.8× |
| Mid-latitude ocean (30–50°) | 0.9× |
| Tropical land | 1.1× |
| Tropical ocean | 0.65× |

**`packages/agents/snapshot.js`** — renders a 1440×720 equirectangular PNG.

Rather than writing to disk (problematic for concurrent users), the agent
encodes the PNG to a Node.js Buffer and emits `{ frame: { year, tick, buf } }`
into the bus. The server tap picks this up and streams it to the client over
socket.io as binary. The client creates a Blob URL and updates `<img src>`.

Disk write still available: set `outputDir` on the agent (or `OUTPUT_DIR` env
var for CLI use) — useful for generating static frames without running the web.

**`public/orbital/ipcc/manifest.js`** — loads elevation + atmosphere + snapshot.
Interval is every 5 ticks (5 years). The scenario is configurable via the
`SCENARIO` env var.

**`public/orbital/ipcc/index.html`** — standalone scenario page served at
`/orbital/ipcc` by Fastify's static catch-all. Connects to socket.io, POSTs
to `/api/sim`, shows live year/CO₂/ΔT stats, and updates the world map as
each frame arrives. Run/Stop controls. No build step.

## Sim lifecycle

**Before this session:** sims ran until `DELETE /api/sim/:id`. No timeouts,
no auto-stop. A user closing a tab left the sim running forever.

**Now:**

- `maxTicks` parameter on `startSim` — tap counts ticks and calls `stopSim`
  via `setImmediate` when the limit is reached, so the final tick and its
  frame render complete before teardown. IPCC page passes `maxTicks: 76`.

- Idle timeout — when a sim's socket room becomes empty (unsubscribe or
  disconnect), a 5-minute timer starts. Subscribing again cancels it. Default
  in `index.js` is `IDLE_MS = 5 * 60 * 1000`.

- The `stopped` event (fired by either path) clears any pending idle timer.

**Memory isolation:** each `createBus()` is its own heap, but all sims share
the same Node.js process and V8 event loop. A CPU-bound sim blocks the others.
No OS-level isolation.

## Elevation raster sharing

**Problem:** each bus instance called `readFile` on the 18MB GEBCO raster,
creating a separate `Int16Array` per concurrent user. At N users = N×18MB.

**Fix:** module-level Promise-based cache in `elevation.js`.

```javascript
const rasterCache = new Map(); // abs path → Promise<{data,width,height}>
```

Storing a Promise (not the resolved value) handles concurrent loads — two buses
requesting the same path both await the same Promise rather than each triggering
a separate `readFile`. On error, the cache entry is deleted so a retry is
possible.

The same pattern is applied to tile set metadata (`tileSetCache`). Individual
tile loading (lazy, per-tile) was already safe because the `tile.loading` flag
prevents duplicate loads on shared tile objects.

The shared `Int16Array` is read-only after initial load — no cross-sim
contamination. Tile objects are mutable (`loading`, `data` flags) but
mutating them is the correct behavior: one bus loading a tile makes it
available to all.

## Previously open issues — now closed

**Memory conservation (June 1 2026 note):**
> "treat read-only data (GEBCO raster, land cover) as process-level singletons
> shared across all sim instances... bus.elevation could be installed once at
> process startup and injected rather than reloaded"

Addressed via module-level Promise cache in `elevation.js` (see above).
The approach taken is slightly different from the injection idea: rather than
installing bus.elevation once at startup, each bus still calls `createElevationService()`
but the underlying data (Int16Array for the raster, tile metadata array for GeoTIFF
tiles) is shared via the module cache. Effect is the same — 18MB loaded once,
referenced by all concurrent buses. The injection approach would have required
changes to the bus API; the cache approach is local to elevation.js.

**Publishing state (June 1 2026 note):**
> "tick events only carry {tick, t, dt}. A scenario page that wants to render
> a heatmap will need field values too. Clean path: an emitter agent in
> @orbital/agents that samples bus.insolation, bus.elevation etc. on each tick
> and fires an { emit: {...} } event — server forwards that to the socket room."

Partially addressed — two separate mechanisms now exist:

1. **Frame streaming**: `snapshot.js` (agents package) renders the full
   temperature grid as a PNG buffer and emits `{ frame: { year, tick, buf } }`
   into the bus. The server tap forwards this as binary to the socket room.
   This is exactly the pattern the note describes — domain logic in the agents
   package, server is a dumb forwarder.

2. **Scalar state on tick**: The server tap in `sims.js` directly reads
   `bus.atmosphere?.co2_ppm()` etc. after each tick and bundles those values
   with the tick event. This is the simpler case but it does leak domain
   knowledge (knowing about `bus.atmosphere`) into the server layer. The cleaner
   design per the note would be an `observe` agent in the agents package that
   emits `{ observe: { co2, deltaT, year } }` events, keeping the "what to
   sample" decision in agents not server. Acceptable for now; worth revisiting
   if more scenarios need different scalar outputs.

## TODOs

### Model

- [ ] **Aerosol forcing offset.** The model runs ~0.3°C warm vs. observations
  because it has no negative aerosol forcing. Adding a flat or trajectory-based
  aerosol term would improve the 2024 starting point accuracy.
  Formula: `F_total = F_CO₂ + F_aerosol`, where `F_aerosol` is negative
  (roughly −0.5 to −1.0 W/m² historically, declining as air quality improves).

- [ ] **Longitude-dependent warming pattern.** Current pattern is latitude
  bands only. AR6 shows additional longitudinal variation — Indian Ocean warms
  faster than Pacific, West African monsoon region differs from East African.
  Would require either a 2D lookup table from CMIP6 ensemble output or
  polynomial fits to the AR6 Atlas data.

- [ ] **Precipitation / moisture.** Temperature anomaly is one axis. Drought,
  flood, and monsoon shift are arguably more consequential. Requires a coupled
  hydrological model.

- [ ] **Methane and other GHGs.** Current model is CO₂-only. Methane is ~25%
  of total forcing. Could add a CH₄ trajectory table and a separate forcing
  term: `F_CH₄ = 0.036 · (√CH₄ − √CH₄₀)`.

### Place-based coupling

- [ ] **City / population agents.** An agent at a fixed (lon, lat) calls
  `bus.atmosphere.temperature_anomaly(lon, lat)` and computes local metrics:
  additional days above wet-bulb 35°C, growing degree day shift, water stress.
  Data sources: WorldPop (gridded population, 100m), SEDAC for vulnerability.

- [ ] **Crop zone agents.** FAO crop calendars + temperature sensitivity curves
  → yield change under each scenario. FAOSTAT has yield and area data.

- [ ] **Exposure computation.** Multiply temperature anomaly grid by WorldPop
  grid → person-days of extreme heat. This is the bridge from physical result
  to policy argument.

### Architecture

- [ ] **Server restart persistence.** Sims are wiped on restart. For long-lived
  sims, would need to serialize bus state. Not a priority until we have
  genuinely persistent scenarios (multi-day or multiplayer).

- [ ] **Process isolation.** All sims share one Node.js process and event loop.
  A CPU-intensive sim blocks all others. Fix: worker threads or child processes
  per sim. Worth doing before opening to public traffic.

- [ ] **Elevation raster sharing across restarts.** The raster is now cached
  in-process but reloaded on server restart. For production: keep the 18MB
  raster as a shared memory segment or load it once into the server and pass
  a reference. Low priority — restart is infrequent.

- [ ] **Scenario-level authorization.** Currently any client can POST
  `/api/sim` with any manifest path. Should restrict to known/approved
  manifests or require auth. No security boundary today.

### Web UI

- [ ] **Scenario selector.** Currently hardcoded to SSP2-4.5 in the HTML.
  Let the user choose before starting the run.

- [ ] **Side-by-side comparison.** Run two scenarios simultaneously and
  show their maps in split view. Requires starting two buses and subscribing
  to two socket rooms.

- [ ] **Decade slider.** After a run completes, replay the stored frames
  with a scrubber. Frames are currently ephemeral (Blob URLs, discarded after
  display). Would need to buffer them client-side.

- [ ] **Legend calibration.** Current color scale runs 0–6°C. Should auto-scale
  or label clearly — SSP5-8.5 produces >10°C local anomaly in the Arctic.
